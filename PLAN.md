## 2. 技術選定（最終判断）

| 領域 | 採用 | 理由・備考 |
|---|---|---|
| フレームワーク | **React 18+ + Vite + TypeScript** | SVG編集画面は状態→表示の写像が複雑。Reactの宣言的SVGと相性◎ |
| 状態管理 | **Zustand**（+ temporal ミドルウェアで Undo/Redo） | zundo で履歴管理が薄く書ける |
| ポリゴン演算 | **clipper2-js（純TSポート）を第一候補** | WASM不要 → GitHub Pages で COOP/COEP 問題ゼロ。オフセット(Round)・union・交差判定すべて対応 |
| 輪郭抽出 | **自前実装を第一候補**（marching squares + Douglas-Peucker、計100行程度）。品質不足なら OpenCV.js にフォールバック | OpenCV.js は WASM 約8MB。必要機能は findContours と erosion の2つだけなので、まず軽量自前実装で Phase 1 検証する |
| erosion（白版） | **自前実装**（ImageData 上のモルフォロジー。2〜10px の erosion は単純処理） | OpenCV 不要のまま行ける見込み |
| PDF書き出し | **pdf-lib** + 低レベルAPIで OCG（Optional Content Groups）を自前注入 | pdf-lib は OCG 非対応だが PDFDict 直接操作で実装可能。リスク高めなので Phase 5 に隔離し、SVG+jsx を先に完成させる |
| デプロイ | GitHub Actions → `vite build` → GitHub Pages | このプロジェクトは「ビルドなし」ルールの対象外（ゲームでなくツール、WASM回避でビルドが合理的） |

### ゲーム規約との差分（明示）
- トロピカ世界観・Press Start 2P は**適用しない**（モックのナチュラル文具系デザインが正）
- モバイル縦画面前提も**適用しない**（デスクトップ専用、`min-width: 1240px`）
- 「ビルド不要」原則も**適用しない**（Vite ビルド + Actions デプロイ）
- 相対パス必須は**適用する**（`vite.config.ts` で `base: './'`）

```
│   │   ├── simplify.ts      ← Douglas-Peucker
│   │   ├── offset.ts        ← clipper2 オフセット＋クロージング
│   │   ├── erosion.ts       ← 白版マスク生成
│   │   ├── collision.ts     ← 膨張ポリゴン交差判定
│   │   └── units.ts         ← mm↔px 変換（px化はここだけ）
│   ├── export/
│   │   ├── svg.ts           ← 3グループSVG書き出し
│   │   ├── pdf-ocg.ts       ← レイヤー付きPDF
│   │   └── jsx-script.ts    ← Illustrator用 .jsx 生成
│   ├── components/
│   │   ├── Header / LeftPanel / Canvas / RightPanel / Footer
│   │   └── SettingsModal
│   └── parts/               ← パーツ定義（寸法メタ＋パスデータ）
├── assets/parts/            ← きなこ提供のテンプレSVG（未着）
└── test/                    ← geometry層のユニットテスト（Vitest）
```

設計原則: **geometry/ は DOM・React に一切依存させない**（Vitest で数値検証できる純関数群にする）。

## 4. フェーズ計画

SPEC.md の Phase 1〜8 を実行順に具体化。各フェーズに完了条件を定義。

### Phase 0: 足場（0.5日相当）
- Vite + React + TS 雛形、テーマCSS変数、Vitest、GitHub Actions（Pages デプロイ）
- **完了条件**: 空アプリが Pages 構成でローカル起動

### Phase 1: コア技術スパイク ★最重要・品質ゲート
- 単体検証ページ（本UIとは別のデバッグ画面）: PNG 1枚投入 → 輪郭抽出 → 簡略化 → +0.5mmオフセット(Round) → クロージング角丸め → SVG重ね表示
- パラメータをスライダーで即時変更できるようにし、髪の毛など複雑形状でギザつき確認
- 自前 marching squares で品質不足なら OpenCV.js へ切替判断（このフェーズ内で決める）
- **完了条件**: きなこが実画像で「このカットラインなら入稿できる」と判断すること。**ここを通過するまで Phase 2 以降に進まない**
- **必要物**: テスト用透過PNG（SPEC §12-4）。届くまではフリー素材で代用

### Phase 2: キャンバスと編集操作
- メイン編集画面のUI移植（モックHTML→Reactコンポーネント化）
- D&D追加、選択、移動（ドラッグ＋矢印キー0.1mm/Shift 1mm）、回転（ハンドル＋15°スナップ＋数値入力）、拡縮（再計算）、複製（結果共有）、削除、Undo/Redo
- 間隔チェック（膨張ポリゴン交差・リアルタイム赤表示・マージン侵食判定）
- ズーム/パン、フッター座標表示
- **完了条件**: 10オブジェクト配置でドラッグ60fps、違反表示がモック通り

### Phase 3: 白版
- erosion 自前実装、水色可視化、レイヤー3トグル
- **完了条件**: 縮小量スライダーで見え方が即時変化、画像縁にカラーはみ出しなし

### Phase 4: SVG書き出し＋.jsx（リリース保険）
- 3つの名前付き `<g>` 単一SVG、白版 #000000 変換、Illustrator変換 .jsx 同梱
- **完了条件**: きなこの Illustrator で開いてレイヤー化できること（実機検証必須）

### Phase 5: レイヤー付きPDF（OCG）
- pdf-lib 低レベルAPIで OCG 注入。カット=ベクター（線幅0.1mm・色設定可）、カラー版/白版=設定DPIラスター
- **完了条件**: Illustrator で開いてレイヤーパネルに3層表示。難航したら v1 は SVG+jsx で出してここは v1.1 送り（リリースブロッカーにしない）

### Phase 6: スタンドパーツ
- パーツライブラリ（タブS/M/L・台座 小/中/大）、タブのドラッグ吸着＋輪郭スライド＋union 一体化、台座（穴付き・印刷あり/なし）、タブ↔穴ペアラベル
- **完了条件**: きなこ提供テンプレ寸法と一致（モックの 8×5/10×6/12×7mm 等は仮値。**実寸データ待ち**）
- ※実寸が届くまでは仮寸法で実装を進め、データ差替えだけで済む構造にする

### Phase 7: 設定・プリセット・プロジェクト保存
- 設定モーダル（モック通り）、localStorage 復元、プリセット保存/切替、プロジェクトJSON保存/読込（画像base64埋め込み）
- **完了条件**: リロードで設定維持、JSON読込で配置完全復元

### Phase 8: デプロイ・QA
- Pages 公開、`/qa` での実ブラウザ検証、メモリ計測（A4・350dpi フル書き出し）
- **完了条件**: 公開URLで入稿データ生成が一通り通る

### 実装順の考え方
- Phase 1 が品質ゲート。**ここで「使える」と確信できるまで他を作らない**（SPEC §10 の方針通り）
- Phase 4（SVG）を Phase 5（PDF）より先に完成させ、OCG が難航しても v1 リリース可能な状態を維持（SPEC §11）
- きなこ実機 Illustrator 検証が必要なのは Phase 4 / 5 / 6 の3回

## 5. リスク対応（SPEC §11 への追補）

| リスク | 計画上の対応 |
|---|---|
| OpenCV.js 8MB WASM のロード負荷 | そもそも使わない方針（自前輪郭抽出＋erosion）。Phase 1 で品質判定し、必要時のみ動的ロードで導入 |
| pdf-lib に OCG API がない | PDFDict 低レベル操作で注入。Phase 4 の SVG+jsx を先に完成させ保険化 |
| ドラッグ中のリアルタイム交差判定が重い | 膨張ポリゴンは配置変更時のみ再計算（形状変更なしの移動は transform のみ）。バウンディングボックス事前フィルタ→精密判定の2段構え |
| 高DPI書き出しのメモリ | 編集中は最大1024px プロキシ、書き出し時のみフル解像度をオブジェクト単位で逐次処理 |
| 拡縮のたびに全パイプライン再計算 | 輪郭は画像px座標系で1回だけ抽出し、mm変換係数だけ更新（オフセット幅は実寸依存なのでオフセット以降のみ再計算） |
